# AGENTS.md

이 저장소는 Spring Boot 기반 게시판 프로젝트이며, 모든 기능 개발은 SDD(Spec-Driven Development)를 따른다.

## 핵심 원칙

- 실제 기능 코드를 작성하기 전에 반드시 명세를 먼저 작성한다.
- 아키텍처는 DDD와 헥사고날 아키텍처를 기본 방향으로 삼는다.
- 구현 전에는 해당 기능의 `spec.md`와 `plan.md`를 검토한다.
- 구현 후에는 해당 기능의 `tasks.md` 체크박스를 갱신한다.
- 문서 색인에 영향을 주는 변경이 있으면 `docs/INDEX.md`를 갱신한다.
- 사용자가 명시적으로 요청하지 않는 한 SDD 문서 없이 기능 코드를 바로 추가하지 않는다.
- SDD 문서, 공통 문서, Swagger 설명은 한글 작성을 기본으로 한다.

## 기능 개발 절차

새 기능은 항상 아래 디렉터리 구조를 먼저 만든다.

```text
specs/{번호}-{기능명}/
  spec.md
  plan.md
  tasks.md
  acceptance.md
```

예시:

```text
specs/001-post-create/
  spec.md
  plan.md
  tasks.md
  acceptance.md
```

## 문서 역할

- `spec.md`: 사용자 요구, 도메인 규칙, 용어, 범위와 비범위를 정의한다.
- `plan.md`: DDD 모델, 헥사고날 포트/어댑터, 패키지 배치, 테스트 전략을 정의한다.
- `tasks.md`: 구현 작업을 체크박스 목록으로 관리한다.
- `acceptance.md`: 인수 조건, 예외 케이스, 완료 판정 기준을 정의한다.

## 구현 전 체크리스트

- [ ] 대상 기능의 `specs/{번호}-{기능명}/spec.md`가 존재한다.
- [ ] 대상 기능의 `specs/{번호}-{기능명}/plan.md`가 존재한다.
- [ ] `spec.md`의 기능 범위와 비범위를 확인했다.
- [ ] `plan.md`의 도메인 모델, 포트, 어댑터, 테스트 전략을 확인했다.
- [ ] 기존 문서(`docs/ARCHITECTURE.md`, `docs/CONVENTIONS.md`, `docs/DOMAIN_GLOSSARY.md`)와 충돌하지 않는다.

## 구현 후 체크리스트

- [ ] 관련 테스트를 추가하거나 갱신했다.
- [ ] 가능한 경우 `./gradlew.bat test`로 검증했다.
- [ ] 완료된 작업을 `tasks.md`에서 체크했다.
- [ ] 새 문서나 주요 결정이 생겼다면 `docs/INDEX.md`를 갱신했다.
- [ ] 아키텍처 결정이 필요한 경우 `docs/adr/`에 ADR을 추가했다.
- [ ] 작업 완료 보고에 추천 `git commit` 메시지를 1개 포함했다.

## 코드 작성 규칙

- 도메인 규칙은 가능한 한 도메인 계층에 둔다.
- 애플리케이션 유스케이스는 입력 포트로 표현한다.
- 외부 의존성(DB, Web, Security, 외부 API)은 어댑터로 격리한다.
- 도메인 계층은 Spring, JPA, Web 프레임워크에 의존하지 않도록 설계한다.
- 컨트롤러는 HTTP 요청/응답 변환에 집중하고 비즈니스 규칙을 담지 않는다.

## 작업 완료 보고 규칙

- 작업 완료 보고에는 변경 요약, 검증 결과, 추천 `git commit` 메시지를 포함한다.
- 추천 커밋 메시지는 Conventional Commits 형식을 사용한다.
- 커밋 타입은 영문을 사용하고, 요약은 한글로 작성한다. 예: `docs: 커밋 메시지 추천 규칙 추가`
- 커밋은 사용자가 명시적으로 요청할 때만 실행한다.
